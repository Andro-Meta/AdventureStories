## BUG-01: cure-with-nothing-to-cure still fires advanceTurn and locks the UI for the wrong reason
**Severity:** High
**File:** `actionHandler.js:1768-1789`
**Class:** combat / ui-desync
**Description:** When the player uses a cure-typed consumable but has no matching status effects, the code sets `consumed = false` and pops "had no effect — it was not consumed", but execution still falls into the `if (otherStatsCure.length === 0 && simpleCureDesc) { ... await advanceTurn(); turnAdvanced = true; }` block. The turn advances, the cooldown ticks down, status ticks fire — yet the UI message tells the user "no effect, not consumed". The block is gated only by `otherStatsCure`/description, not by whether anything was actually cured.
**Repro:** Equip a "Cure Poison" potion, use it while uninjured/uncursed.
**Fix:** Wrap the simple-cure advance branch in `if (consumed) { ... }`, or `return` early in the `else` branch above it. Same applies symmetrically to the heal branch when HP is already full (it returns "HP already full" but never advances the turn — inverse issue).
**Effort:** trivial

## BUG-02: Status-effect duration is decremented twice per combat round
**Severity:** High
**File:** `combat.js:991-1135` (processStatusEffectTicks) and `combat.js:595-635` (processRoundEffects)
**Class:** combat
**Description:** `advanceCombatTurn` calls `processStatusEffectTicks(currentTurnChar)` for the character whose turn just ended, which decrements `effect.duration--`. When the round wraps, the same function calls `processRoundEffects()`, which iterates `gameState.players + gameState.enemies` and again does `status.duration--` on every status effect of every combatant. So in a 1-on-1 fight the player's "Poison(4)" expires after 2 rounds, not 4. Worse, `processStatusEffectTicks` is also re-called from `advanceTurn` (turnManager.js:38-49) when combat ends mid-turn — third decrement.
**Repro:** Apply Poison{duration:4} to player; observe expiration in 2 in-combat rounds.
**Fix:** Make `processRoundEffects` only handle round-scoped global effects (the `gameState.combat.activeEffects` strip) and stop touching per-character `statusEffects` durations.
**Effort:** small

## BUG-03: Status effect onTick / onExpire / onRound functions don't survive save/load
**Severity:** High
**File:** `combat.js:991-1135`, `saveLoad.js:84-145`
**Class:** save-load / combat
**Description:** `processStatusEffectTicks` invokes `effect.onTick(character)`, `effect.onExpire(character)`; `processRoundEffects` invokes `status.onRound()`. Effects are stored on `character.statusEffects` and serialized via `JSON.parse(JSON.stringify(gameState))` at save time — function-typed properties drop silently. Loading a saved game with active Poison/Burn/etc therefore stops dealing damage-per-turn (the effectTickData hpPerTurn path still works, but custom onTick effects, Sleep break-on-damage, and custom round handlers do not).
**Repro:** Save mid-combat with a Sleep effect on enemy. Load. Hit it. `Sleep` no longer breaks because its onTick comparator is gone.
**Fix:** Don't store function references on saveable objects. Look up the catalog entry by name on each tick (engine.js already has `lookupStatusEffectCatalog`). Or strip function fields before serialize and re-attach from catalog on load.
**Effort:** medium

## BUG-05: localAI.makeRequest's `||` defaulting overrides explicit zero / falsy values
**Severity:** Medium
**File:** `localAI.js:296-310`
**Class:** backend
**Description:** Code reads `temperature: options.temperature || defaults.temperature`, etc. If a caller passes `temperature: 0` (deterministic) or `top_p: 0`, the falsy `||` evaluates to defaults instead of the requested value. Same bug for `max_tokens`, `top_p`, `top_k`. The narrative-turn JSON path explicitly tunes these (e.g. choices use `max_tokens: 512`); a future caller wanting low-temperature deterministic output for tests/audits will silently get default 0.7.
**Repro:** Call `getAIResponseJSON(msgs, schema, { temperature: 0, max_tokens: 0 })`. Both fall through to defaults.
**Fix:** Use `options.temperature ?? defaults.temperature` (nullish coalescing) on all four lines.
**Effort:** trivial

## BUG-06: Unconditional retry on every error type, including HTTP 4xx
**Severity:** Medium
**File:** `localAI.js:445-460` (executeRequest)
**Class:** backend
**Description:** The catch block retries up to `MAX_RETRIES` for any error — including HTTP 400 "invalid request body" or 401 "unauthorized" (cloud bad API key). Each retry waits `RETRY_DELAY * (attempt+1)` ms, so a malformed schema spends ~6+ seconds retrying before the user sees the failure. Worse, on cloud backends, repeated 401 retries can trigger rate-limit/abuse counters.
**Repro:** Configure cloud backend with invalid API key. Submit any choice. Wait 8-10s instead of <1s for the auth error to surface.
**Fix:** Only retry for network-level errors (`AbortError`, `TypeError: Failed to fetch`) and 5xx. For 4xx responses, throw immediately.
**Effort:** small

## BUG-08: God-mode UI doesn't render after load because isActive is always false on the freshly-attached singleton
**Severity:** High
**File:** `saveLoad.js:296-302`, `godMode.js:18-32`, `ui.js:1031`
**Class:** god-mode / ui-desync / save-load
**Description:** The `GodModeManager` instance is re-imported via `await import('./godMode.js')` on load and re-attached to `gameState.godModeManager`. The constructor sets `this.isActive = false`, `this.isUnlocked = false`. The save data only stored the JSON form of the previous instance, which Object.assign overwrote into `gameState.godModeManager` BEFORE the re-import overwrote it back to a fresh class. **Repro:** Unlock god mode, save, reload page, load the save. No god-mode input box.
**Fix:** After re-attaching the singleton in saveLoad.js, run `if (gameState.isGoalComplete) gameState.godModeManager.activateGodMode()`. The engine's `/isGoalComplete` apply path already does this on transition; do it on load too.
**Effort:** trivial

## BUG-09: Pre-applied god-mode coin op clobbers existing wealth instead of summing
**Severity:** Medium
**File:** `actionHandler.js:973-987`, `engine.js:80-95`
**Class:** god-mode
**Description:** `extractGodModeDiffOps` matches "I have a million gold" → emits `replace /players/0/coins value 99999`. The engine's coins handler does `player.coins = Math.max(0, value)` — it overwrites. So if the player already had 50,000 coins and types "I gain a hundred gold", the regex captures `100`, the op REPLACES coins with 100, and they LOSE 49,900 coins. The verb in the regex is "(have|gain|get|now have)" — the gain path should add, not replace.
**Repro:** With 50000 coins, type "I gain a hundred gold". End up at 100.
**Fix:** Distinguish "have/now have" (replace) from "gain/get" (add). For gain verbs, look up current coins and emit `replace` with sum, capped at 99999.
**Effort:** small

## BUG-10: Pre-applied god-mode item op auto-equips unconditionally, kicking out previously-equipped legendary items
**Severity:** Medium
**File:** `actionHandler.js:1052-1063`
**Class:** god-mode
**Description:** Any "I wield/wear" pattern triggers an unconditional `replace /players/0/equipment/{weapon,armor}` to the new item id. The engine's equipment handler `apply` clears the previously-equipped item's `equippedSlot`. The player has no way to say "I have a Mythril Dagger in my pack" without auto-replacing whatever's currently in the weapon slot. Easy to lose a high-stat earned weapon to a flavor declaration.
**Repro:** With Excalibur equipped (atk:80), type "I have a small dagger." Excalibur unequips, dagger equips with default atk:24.
**Fix:** Only auto-equip if the player's current weapon slot is empty, OR if the new item's stats are >= current. Better: have the regex distinguish "I have/carry" (don't equip) from "I wield/swap to" (equip).
**Effort:** small

## BUG-30: handlePlayerChoice's "FINAL SAFETY NET" fallback re-renders choices but doesn't clear isLoading flag — AI-still-processing state can persist
**Severity:** Low
**File:** `actionHandler.js:621-664`
**Class:** ui-desync
**Description:** The finally block does `UI.showLoading(false)` then renders fallback choices. **Effort:** trivial
